Poddy integrates deeply with Zendesk to streamline support workflows. Staff can escalate user issues from Discord directly into RunPod's support ticketing system, preserving context and attachments.

## How Zendesk escalation works

When a user needs help beyond what Discord can provide, staff members can escalate messages or entire threads to create Zendesk support tickets automatically.

### Escalating a message

Staff can escalate individual messages to Zendesk:

1. Click the "Escalate to Zendesk" button on a message
2. Poddy checks if the user has a linked RunPod account via GraphQL
3. If found, the email is auto-filled; otherwise, staff manually enters it
4. A Zendesk ticket is created with the message content and attachments

<CodeGroup>

```typescript src/utilities/functions.ts
export default class PoddyFunctions extends Functions {
  public async submitTicket(
    type: "message" | "thread",
    email: string | undefined,
    user: APIUser,
    escalatedId: string,
    interaction: APIModalSubmitGuildInteraction,
    ticket: SubmittableTicket,
  ) {
    // Check for existing ticket to prevent duplicates
    const existingTicket = await this.client.prisma.zendeskTicket.findUnique({
      where: { escalatedId },
    });

    if (existingTicket) {
      // Already escalated - notify user
      return null;
    }

    // Create Zendesk ticket via API
    const response = await fetch("https://runpodinc.zendesk.com/api/v2/tickets.json", {
      headers: {
        "Content-Type": "application/json",
        Authorization: `Basic ${env.ZENDESK_API_KEY}`,
      },
      body: JSON.stringify({
        ticket: {
          subject: ticket.subject ?? `${type[0]?.toUpperCase() + type.slice(1)} Escalated From Discord`,
          comment: ticket.comment,
          requester: { email, name: interaction.member.user.username },
          tags: ["discord"],
        },
      }),
      method: "POST",
    });

    const data: ZendeskCreateTicketResponse = await response.json();

    // Store ticket mapping in database
    await this.client.prisma.zendeskTicket.createMany({
      data: [{
        id: data.ticket.id,
        escalatedId,
        escalatedById: user.id,
        type: type === "message" ? EscalatedType.MESSAGE : EscalatedType.THREAD,
      }],
      skipDuplicates: true,
    });

    return data;
  }
}
```
</CodeGroup>

### Escalating a thread

Thread escalation captures the full conversation history:

```typescript src/bot/modals/zendesk/escalateToZendesk.ts
const data = await this.client.functions.submitTicket(type, email, user, id, interaction as any, {
  comment: {
    html_body: `${user.username} [${user.id}] escalated 
    <a href="https://discord.com/channels/${thread.guild_id!}/${thread.id}">this thread</a>
    from Discord, all messages in the thread are included below as internal notes.`,
  },
});

if (!data) return;

// Fetch all thread messages (excluding bots)
const allMessages = await this.client.functions.fetchAllThreadMessages(thread.id, {
  excludeBots: true,
  sortOrder: "chronological",
});

// Add each message as internal note to the ticket
for (const message of allMessages) {
  await this.client.functions.addTicketInternalNote(data.ticket.id, message);
}
```

<Accordion title="How thread messages are fetched">
Poddy uses pagination to fetch the complete thread history:

```typescript src/utilities/functions.ts
public async fetchAllThreadMessages(
  threadId: string,
  options: {
    excludeBots?: boolean;
    limit?: number;
    sortOrder?: "chronological" | "reverse";
  } = {},
) {
  const { excludeBots = false, limit, sortOrder = "chronological" } = options;
  const allMessages: APIMessage[] = [];

  let messages = await this.client.api.channels.getMessages(threadId, {
    limit: 100,
  });

  allMessages.push(...messages.filter((message) => (excludeBots ? !message.author.bot : true)));

  // Paginate through all messages
  while (messages.length === 100 && (!limit || allMessages.length < limit)) {
    messages = await this.client.api.channels.getMessages(threadId, {
      limit: 100,
      before: messages[messages.length - 1]!.id,
    });

    const filteredMessages = messages.filter((message) => (excludeBots ? !message.author.bot : true));

    if (limit && allMessages.length + filteredMessages.length > limit) {
      const remaining = limit - allMessages.length;
      allMessages.push(...filteredMessages.slice(0, remaining));
      break;
    }

    allMessages.push(...filteredMessages);
  }

  if (sortOrder === "chronological") {
    allMessages.reverse();
  }

  return allMessages;
}
```
</Accordion>

## Attachment handling

Discord attachments are uploaded to Zendesk before ticket creation:

```typescript src/utilities/functions.ts
public async uploadAttachmentsToZendesk(
  attachments: { url: string; filename: string; content_type?: string | null }[],
): Promise<string[]> {
  return Promise.all(
    attachments.map(async (attachment) => {
      const buffer = await fetch(attachment.url).then(async (res) => res.arrayBuffer());

      const response = await fetch(
        `https://runpodinc.zendesk.com/api/v2/uploads.json?filename=${attachment.filename}`,
        {
          headers: {
            Authorization: `Basic ${env.ZENDESK_API_KEY}`,
            "Content-Type": attachment.content_type ?? "application/octet-stream",
          },
          body: buffer,
          method: "POST",
        },
      );

      const data: ZendeskUploadResponse = await response.json();
      return data.upload.token;
    }),
  );
}
```

These upload tokens are then attached to the ticket comment.

## Database schema

Zendesk tickets are tracked to prevent duplicate escalations:

```prisma prisma/schema.prisma
enum EscalatedType {
  MESSAGE
  THREAD
}

model ZendeskTicket {
  id            Int           @id
  escalatedId   String        @unique  // Message or thread ID
  escalatedById String                 // Staff member who escalated
  type          EscalatedType

  @@map("zendesk_tickets")
}
```

<Info>
The `escalatedId` unique constraint ensures each message or thread can only be escalated once, preventing duplicate tickets.
</Info>

## User workflow

<Steps>
  <Step title="User asks for help in Discord">
    A user posts a question or problem in a support channel or creates a thread.
  </Step>
  
  <Step title="Staff identifies escalation need">
    Staff determines the issue requires formal support ticket handling.
  </Step>
  
  <Step title="Staff clicks escalate button">
    Staff clicks the "Escalate to Zendesk" button on the message or thread.
  </Step>
  
  <Step title="Email lookup or entry">
    Poddy checks for a linked RunPod account. If not found, staff enters the user's email.
  </Step>
  
  <Step title="Ticket created in Zendesk">
    A support ticket is created with the Discord context, attachments, and thread history.
  </Step>
  
  <Step title="Database record prevents duplicates">
    The ticket is stored in the database to prevent re-escalation of the same issue.
  </Step>
</Steps>

## Error handling

Poddy includes robust error handling with Sentry integration:

```typescript
if (response.status !== 201) {
  const eventId = await this.client.logger.sentry.captureWithExtras(
    new Error(`Zendesk Ticket Creation Failed With Status ${response.status} (${response.statusText})`),
    { response },
  );

  await this.client.api.interactions.editReply(interaction.application_id, interaction.token, {
    embeds: [{
      title: lang.get("ESCALATED_TO_ZENDESK_ERROR_TITLE"),
      description: lang.get("ESCALATED_TO_ZENDESK_ERROR_DESCRIPTION"),
      footer: { text: lang.get("SENTRY_EVENT_ID_FOOTER", { eventId }) },
      color: botConfig.colors.error,
    }],
    flags: MessageFlags.Ephemeral,
  });
  return null;
}
```

All errors are logged to Sentry with full context for debugging.
